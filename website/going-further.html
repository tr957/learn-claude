Going Further

Going Further with Claude

Beyond one-off prompts — how to use Claude across projects, repeat workflows reliably, and save yourself time on recurring work.

Claude chat vs Claude Code — which to use when

Most people using Claude are using it in a chat window: you type a prompt, Claude responds, you move on. That covers the majority of tasks.

Claude Code is different. Instead of chatting, Claude can open your actual files, make changes across a project, and run tasks end to end — without you having to copy and paste anything. Think of it as the difference between describing a problem to someone versus sitting them down at your computer.

Neither is better. They are for different things.

Use normal Claude chat when you are asking a question, drafting something, or working through a one-off task. Most of what is on this site falls into this category.
Use Claude Code when you are editing files, running a task that touches multiple documents, or you need Claude to work through something that would take too many copy-paste cycles in chat.

Real example: This website was built and updated using Claude Code. When a search bar needed adding to every page, Claude Code made the change across all ten pages in a single session. In chat, that would have been ten manual edits.

Working across a project, not just a single prompt

A single prompt treats Claude like a search engine: one question, one answer. A project works differently. You give Claude the full context — background documents, meeting notes, investment memos, founder materials — and it holds onto all of it. Then you can ask questions, make requests, and move through a task without re-explaining anything.

Use this when you are doing deep diligence on a company and want Claude to reference the whitepaper, the founder interview transcript, and your own notes all at once — or when you are writing a memo and want Claude to stay consistent with decisions made earlier in the process.

Load a founder's deck, their whitepaper, and notes from three calls into a project. Then ask:

"Based on everything here, what claims in the deck are not supported by — or directly contradict — the technical documentation?"

That kind of synthesis is only possible when Claude has access to all the material at once. In a standard chat, you would have to manually compare them yourself.

Turning repeated work into a reusable skill

Every time you start a new chat, Claude starts fresh. It does not know your preferences, your format, or your context. If you run the same type of task regularly — meeting summaries, diligence briefs, weekly updates — you are re-explaining yourself every time.

A reusable skill fixes this. You write down your instructions once: what the task is, what format you want, what context Claude should always have. Every time you run that task, Claude already knows the rules.

Use this when you have run the same type of prompt more than three times and found yourself typing similar setup instructions at the start of each conversation.

Here is what a meeting summary skill looks like in practice. Write this once and save it — then paste your raw notes and Claude produces the same structured output every time:

"When I give you notes from a founder call, always produce the following: one paragraph of context on the company, the three most important claims the founder made, two open questions that were not answered, and a recommended next step. Keep the whole thing under 300 words."

From one-off prompt to repeatable workflow

Most good workflows start as a single rough prompt. Here is how that evolves — using a pre-investment deck review as the example:

Stage 1 — The rough ask

"Summarise this deck." You get something back. It is useful. You move on.

Stage 2 — Improving the prompt

You run it a few more times and realise you always want the same things: token model, team background, competitive risks. You refine it:

"Summarise this deck. Focus on: token economics, founding team credentials, and the three strongest competitive risks. Format as three short sections."
Stage 3 — You are using it regularly

You run this before every investment call. It saves 20 minutes of prep each time.

Stage 4 — You save it as a skill

You write it down properly — with context about 1kx's investment focus, the format you want, and what to skip. Now anyone on the team can use it without you sharing the prompt each time.

Stage 5 — It becomes a workflow

You chain it with two more steps: after the summary comes a red-team pass ("argue against this investment"), then a gap list ("what questions are still unanswered"). One task, three outputs, run the same way every time.

The jump from stage 1 to stage 5 does not require technical skills. It requires noticing that you are doing the same thing repeatedly, and deciding it is worth making consistent.

Editing multiple files at once

In a chat window, you work on one thing at a time. You paste something in, Claude responds, you take the output and apply it manually. If you have ten documents that all need the same change, that is ten separate conversations — or ten copy-paste cycles.

Claude Code can work across multiple files in one session. You point it at a folder, describe what needs to change, and it makes the change everywhere consistently.

Use this when you need to update a template that exists across multiple documents, apply a consistent formatting change, or make an edit that touches more than two or three files.

Real example: Updating the navigation bar on this website required changes to ten HTML files. In Claude Code, that was a single instruction. In chat, it would have required manually editing each file and checking each one for consistency.

When to stay simple vs when to build a workflow

Not everything needs to be a workflow. The honest answer is that most tasks do not. Here is a rough guide:

Stick with normal chat when you are asking a one-off question, drafting something you will not repeat, or the task fits comfortably in a single conversation. This is most things.
Think about Claude Code when you are editing actual files on your computer, the task touches more than two or three documents, or you have been copy-pasting Claude's output into files manually more than a few times.
Think about a reusable skill when you have run the same type of prompt three or more times, you keep re-typing the same setup instructions, or other people on the team would benefit from using the same consistent format.
Think about a full workflow when you have a multi-step process you run on a regular cadence — per deal, per meeting, per week — and getting it right consistently would save real time or reduce the chance of missing something.

The goal is not to automate everything. It is to notice when you are doing repetitive manual work that Claude could handle reliably, and make that investment once.

What this makes possible at 1kx

Put together, these features shift what Claude can do for the team beyond answering individual questions:

  • A pre-deal workflow that takes a deck from raw input to summarised brief, red-teamed thesis, and open question list — in a consistent format, every time, regardless of who runs it
  • A post-meeting skill that produces a clean summary with key claims, follow-up questions, and a recommended next step from raw notes — without anyone having to write it up from scratch
  • A cross-document analysis that runs across a founder's deck, whitepaper, and interview transcripts at once and flags where the story does not hold together
  • Claude Code for operations — updating templates, maintaining consistent documents, making changes across files without manual copy-paste

None of this requires engineering skills. It requires noticing which parts of your work are repetitive and deciding it is worth spending an hour setting them up properly.
